O Conceito de Banco de Dados (BD)

O que é um Banco de Dados (BD)?

Um Banco de Dados é uma coleção organizada de dados inter-relacionados.

  • Ele permite o armazenamento, gerenciamento e recuperação eficiente de informações.
  • Propósito: Transformar dados brutos em informação significativa para a tomada de decisões.

Imagine-o como um grande arquivo digital, mas com regras rigorosas para manter a ordem e a integridade.

Tipos de Bancos de Dados

Principais tipos

Tipo Característica Principal Exemplo
Relacional (SQL) Estrutura rígida (tabelas); Foco em relações e integridade. MySQL, PostgreSQL, Oracle, SQL Server
Não Relacional (NoSQL) Estrutura flexível (documentos, grafos, chave-valor); Foco em escalabilidade e velocidade. MongoDB, Redis, Neo4j

Nossa disciplina se concentra nos BDs Relacionais, especificamente o MySQL.

O Modelo Relacional (MR)

Entendendo a Base do MySQL

O Modelo Relacional (MR) armazena dados em tabelas (também chamadas de relações). As relações são interconectadas por meio de valores em campos comuns (chaves).

Vantagens do MR Desvantagens Comuns
Alta Integridade e Consistência. Menor escalabilidade horizontal (tradicionalmente).
Fácil de entender e gerenciar. Rigidez de esquema.
Uso da linguagem padronizada SQL. Performance em dados massivos e não estruturados.

Sistema de Gerenciamento de BD (SGBD)

O Guardião dos Dados (Ex: MySQL)

O SGBD (Sistema de Gerenciamento de Banco de Dados) é o software que interage com o usuário, a aplicação e o banco de dados.

Funções Chave:

  • Definir e Modificar a Estrutura: Linguagem DDL (Data Definition Language).
  • Manipular os Dados: Inserir, Atualizar, Excluir (Linguagem DML).
  • Garantir a Governança: Segurança, concorrência e integridade dos dados.

Componentes Essenciais: Tabela (Table)

A Unidade Fundamental de Armazenamento

  • Uma Tabela (ou Relação) representa uma única entidade (Ex: Cliente) ou um relacionamento.
  • É organizada em Colunas (atributos) e Linhas (registros).
  • Nomeação: Deve ser singular e descritivo (Ex: Clientes, Produtos, Pedidos).

Tabela: `Clientes` (Estrutura)

ID_Cliente, Nome, Sobrenome, Email, Data_Cadastro

Componentes Essenciais: Coluna (Column)

Os Atributos da Entidade

  • Uma Coluna (ou Atributo) define uma peça específica de informação.
  • Cada coluna tem um nome (Ex: Email) e um tipo de dado (Ex: VARCHAR(100)).
  • As colunas definem a estrutura (o esquema) da tabela.

Exemplo na tabela `Clientes`:

`Email` (VARCHAR), `ID_Cliente` (INT), `Data_Cadastro` (DATE)

Componentes Essenciais: Linha (Row)

O Registro de Dados

  • Uma Linha (ou Tupla ou Registro) é o conjunto completo de valores para um único item.
  • Representa uma ocorrência individual da entidade (Ex: um cliente específico).

Linha de Exemplo:

ID_Cliente (PK) Nome Email
1 Maria maria@email.com

Integridade de Dados

A Qualidade e Confiabilidade da Informação

Integridade de Dados refere-se à exatidão, consistência e confiabilidade dos dados no BD.

Por que é vital?

  • Evita erros de lógica na aplicação (Ex: falhas no carrinho de compras).
  • Garante que relatórios e análises sejam precisos.
  • Exemplo de Problema: Vender um produto com estoque negativo, resultando em dados inconsistentes.

O uso de Chaves (PKs e FKs) é a principal ferramenta para garantir essa integridade.

Chaves (Keys)

O Mecanismo de Conexão e Identificação

Chaves são colunas usadas para identificar linhas de forma única e conectar tabelas entre si.

1. Chave Primária (Primary Key - PK):
Identifica a linha unicamente dentro da tabela.

2. Chave Estrangeira (Foreign Key - FK):
Garante a integridade referencial entre tabelas.

Chave Primária (PK)

O Identificador Único da Tabela

A Chave Primária (PK) é a coluna que identifica cada linha de forma única na tabela.

Regras Essenciais:

  • Unicidade: Nenhum valor pode se repetir (Ex: Dois clientes não podem ter o mesmo ID).
  • Não Nula: O valor deve sempre existir (não pode ser NULL).

Frequentemente, é um número sequencial e automático (auto-incremento), como o ID_Cliente.

Chave Estrangeira (FK)

O Vínculo entre Tabelas

A Chave Estrangeira (FK) é uma coluna na tabela filha que referencia a PK da tabela mãe.

Exemplo Visual:

Tabela Mãe: CLIENTES (PK: ID_Cliente)
↓ Referência
Tabela Filha: PEDIDOS (FK: ID_Cliente)

Propósito: Criar e impor o relacionamento "um-para-muitos" (Um cliente pode ter muitos pedidos).

Integridade Referencial

Mantendo a Consistência dos Relacionamentos

É a regra imposta pela Chave Estrangeira (FK) para garantir que as referências sejam sempre válidas.

A Integridade Referencial Garante Que:

  • Você não pode inserir um valor na tabela filha (FK) que não existe na tabela mãe (PK).
  • Exemplo Prático: É impossível registrar um Pedido com um ID_Cliente que não está cadastrado na tabela Clientes.

Isso evita "registros órfãos" e mantém o BD logicamente consistente.

Tipos de Dados (Data Types)

Definindo o Tipo de Conteúdo de uma Coluna

O Tipo de Dado especifica a natureza da informação armazenada (Ex: Número, Texto, Data).

Por que a escolha é crucial?

  • 1. Integridade: Garante que os dados sejam válidos (Ex: não permite texto em um campo numérico).
  • 2. Performance: Otimiza o armazenamento, pois um tipo menor consome menos espaço, e acelera as consultas.

Tipos de Dados Comuns: Numéricos

Valores de Contagem e Cálculo

Tipo (MySQL) Descrição Exemplo de Uso
INT Inteiro de tamanho padrão. `ID_Cliente`, `Quantidade_Estoque`
DECIMAL Armazenamento exato de ponto flutuante (Ideal para moedas). `Preco_Unitario` (Ex: `DECIMAL(10, 2)`)
FLOAT / DOUBLE Armazenamento aproximado de ponto flutuante. `Latitude`, `Longitude`

Tipos de Dados Comuns: Strings

Valores de Texto

Tipo (MySQL) Descrição Exemplo de Uso
VARCHAR Strings de tamanho variável. Mais comum e eficiente. `Nome_Produto`, `Endereco`, `Email`
CHAR Strings de tamanho fixo. Preenche com espaços. Códigos de país (Ex: `CHAR(3)`)
TEXT Armazena grandes volumes de texto (até 64KB). `Descricao_Longa_Produto`

Dica: Use VARCHAR para textos onde o tamanho varia para economizar espaço de armazenamento.

Tipos de Dados Comuns: Data e Hora

Valores Temporais

Tipo (MySQL) Descrição Formato
DATE Apenas a data. `YYYY-MM-DD`
TIME Apenas a hora. `HH:MM:SS`
DATETIME Data e hora combinadas. `YYYY-MM-DD HH:MM:SS`
TIMESTAMP Útil para registrar atualizações automaticamente. `YYYY-MM-DD HH:MM:SS`

Breve Introdução ao SQL

A Linguagem de Consulta Estruturada

O SQL (Structured Query Language) é a linguagem universal para interagir com BDs Relacionais.

As Quatro Ações Fundamentais (CRUD):

C(riar) → INSERT

R(er) → SELECT

U(pdate) → UPDATE

D(elete) → DELETE

Na ultima aula, usamos no comando SELECT para recuperar dados.

Resumo

Revisão Rápida:

  • A estrutura é baseada em Tabelas, Colunas e Linhas.
  • A PK garante a identificação única.
  • A FK garante a integridade dos relacionamentos.

Boas Práticas: Nomenclatura

Deixando o Código Limpo e Consistente

Regras Comuns (Snake_Case):

Tabelas: Use o nome no Plural.

Ex: `clientes`, `produtos` (não `cliente`, `produto_tabela`).

Colunas: Use o nome no Singular e letras minúsculas (snake_case).

Ex: `nome`, `email_cliente` (não `NomeCliente`).

Chaves (PK/FK): Use prefixos claros.

Ex: `id_cliente` (PK), `cliente_id` (FK - Alternativa comum).

Escolhendo Chaves Primárias

Substitutas vs. Naturais

1. Chaves Substitutas (Surrogate Keys)

  • Valor artificial (geralmente um INT autoincremental).
  • Vantagem: Simples, nunca muda, pequeno, rápido.
  • Desvantagem: Não tem significado para o usuário final.

2. Chaves Naturais

  • Valor baseado em dados reais (Ex: CPF, CNPJ, Número de Série).
  • Risco: Podem ser longas, complexas (requerem chaves compostas) ou, pior, podem MUDAR (Ex: formato de CPF em outro país).

Recomendação: No modelo físico (SQL), sempre que possível, utilize Chaves Substitutas (IDs simples). São mais práticas.

Restrições de Coluna (Constraints)

Controlando a Obrigatoriedade e Unicidade

Restrição Propósito Exemplo
NOT NULL Obrigatório. O campo deve ter um valor. `nome`, `data_nascimento`
NULL (Padrão) Opcional. Pode ser deixado em branco. `telefone_opcional`, `observacoes`
UNIQUE Garante que não haverá valores duplicados nesta coluna. `email`, `cpf` (se for natural key)
DEFAULT Define um valor padrão se nenhum for fornecido. `status_ativo` = 1

Tipos de Relacionamento

A Estrutura do Modelo Conceitual

1. Um para Um (1:1)

Cada registro em A se relaciona com, no máximo, um registro em B. (Ex: Pessoa e Passaporte).

2. Um para Muitos (1:N)

Mais Comum. Um registro em A pode ter muitos registros em B.

Ex: 1 Cliente para N Pedidos. (Implementado via FK).

3. Muitos para Muitos (N:M)

Um registro em A pode ter muitos em B, e vice-versa.

Solução: Requer uma terceira tabela de Junção ou Ligação (Ex: Pedidos_Produtos).

O Ciclo de Design do BD

Design e Implementação

1. Requisitos: Entender o que o BD precisa fazer e quais dados armazenar.
2. Modelo Conceitual (ER): Mapear Entidades, Atributos e Relacionamentos.
3. Modelo Lógico: Definir Tabelas, Colunas, Chaves e Tipos de Dados.
4. Modelo Físico: Implementação concreta no SGBD (Ex: Usando SQL para CREATE TABLE).

Ao traduzir nossos modelos para SQL, já estamos trabalhando com o modelo físico.

Exemplo Prático 1: Criando o Banco de Dados

DDL: CREATE DATABASE

Comando para criar um novo banco de dados (esquema). O `IF NOT EXISTS` evita erros caso o BD já exista.

-- Comando para Criar o Banco de Dados (Schema) CREATE DATABASE IF NOT EXISTS meu_primeiro_bd; -- Comando para selecionar o BD como o atual USE meu_primeiro_bd;

No Workbench: Digite e execute os comandos em um painel de consulta. O BD aparecerá na lista de Schemas.

Exemplo Prático 2: Criando a Tabela 'clientes'

DDL: CREATE TABLE (Definindo a PK)

Criamos a tabela `clientes` com a Chave Primária (PK) autoincremental e restrições `NOT NULL`.

-- Cria a tabela de clientes dentro do BD selecionado CREATE TABLE clientes ( id_cliente INT NOT NULL PRIMARY KEY AUTO_INCREMENT, nome VARCHAR(100) NOT NULL, email VARCHAR(100) UNIQUE, data_cadastro DATE );

Atenção: AUTO_INCREMENT só funciona em colunas de chave.

Exemplo Prático 3: Visualizando a Estrutura

SQL: SHOW TABLES e DESCRIBE

Comandos de utilidade (DDL) para verificar o conteúdo do BD e a estrutura da tabela, respectivamente.

Visualizar Tabelas (Dentro do BD atual):

SHOW TABLES; -- Resultado: | Tables_in_meu_primeiro_bd | -- | clientes |

Descrever Estrutura (Metadados):

DESCRIBE clientes; -- Resultado mostra: Field, Type, Null (NO/YES), Key (PRI), Default, Extra (auto_increment)

Exemplo Prático 4: Inserindo Dados

DML: INSERT INTO (Adicionando Registros)

O primeiro comando DML. Insere novos registros (linhas/tuplas) na tabela. Note a omissão do `id_cliente` (é auto-incrementado).

-- Inserindo um novo cliente na tabela 'clientes' INSERT INTO clientes (nome, email, data_cadastro) VALUES ( 'Joao Silva', 'joao.silva@exemplo.com', '2025-01-10' );

Próxima Prática: Insira mais 2 ou 3 registros para testar!

Exemplo Prático 5: Excluindo Estruturas

DDL: DROP TABLE / DATABASE (Cuidado!)

Use estes comandos com extrema cautela, pois eles removem a estrutura e todos os dados de forma permanente!

Remover uma Tabela (DDL):

DROP TABLE IF EXISTS clientes;

Remover o Banco de Dados (DDL):

DROP DATABASE IF EXISTS meu_primeiro_bd;
